# Общее описание заданий

## О проекте

Веб-приложение — аналитический дашборд, который визуализирует статистику пользователей игровой платформы: графики, таблицы, карточки с метриками, история событий, сравнительный анализ. Приложение поддерживает светлую и тёмную тему, адаптировано под мобильные устройства.

### Текущее состояние приложения

- Приложение состоит из нескольких разделов (табов), между которыми пользователь переключается
- Адаптивная вёрстка — интерфейс корректно отображается на мобильных устройствах
- Приложение поддерживает **тёмную и светлую тему** (переключатель в шапке, сохраняется в localStorage)
- Данные предзагружены в HTML-файл (статическая генерация) — приложение не делает запросов к серверу

### Стек технологий

- **React** — JavaScript-библиотека для построения пользовательских интерфейсов на основе компонентов
- **TypeScript** — надмножество JavaScript с системой типов, помогает избежать ошибок на этапе разработки
- **Vite** — современный инструмент сборки для веб-приложений, обеспечивает быструю разработку (hot reload) и оптимизированную сборку
- **Tailwind CSS** — CSS-фреймворк утилитарных классов, позволяет стилизовать элементы прямо в разметке без написания отдельных CSS-файлов
- **ECharts** — библиотека для создания интерактивных графиков и диаграмм

### Как устроена будет работа студентов

- Каркас приложения и архитектура уже подготовлены
- В коде оставлены комментарии `// TODO:` с описанием того, что нужно реализовать
- Каждый `TODO` содержит: задачу, подсказки по реализации, ссылки на аналогичные места в коде
- Для продвинутых студентов — дополнительные задания, помеченные `// BONUS:`
- Студенту не нужно создавать файлы с нуля или настраивать архитектуру — нужно дописать логику в подготовленных местах

### Что появится в приложении после выполнения всех заданий

- Новые визуализации: радарная диаграмма, раскрываемые карточки, блок лучших/худших результатов
- Динамическая загрузка данных через API (вместо предзагруженных данных)
- Поиск пользователей в реальном времени с оптимизацией запросов
- Сравнение двух пользователей (side-by-side)
- Автообновление данных с визуальным индикатором
- Анимации, loading-скелетоны, тематические изображения, SVG-иконки
- Полное покрытие тестами (unit + интеграционные)
- REST API proxy-сервер для безопасного взаимодействия с внешним API

**Ожидаемый результат**: работающий таб с интерактивной радарной диаграммой, где каждая ось отображает одну из метрик, а каждый элемент — отдельный многоугольник на графике.

**Ожидаемый результат**: интерактивная таблица, где клик по строке раскрывает карточку с детальной информацией, повторный клик — сворачивает.

**Ожидаемый результат**: секция из 4 карточек на странице обзора — лучший и худший элемент по проценту побед, лучший по эффективности и лучший по урону.

### Тема 3 — Реализация и внедрение RESTful API для обеспечения взаимодействия между клиентом и сервером

#### Задание 3.1 — REST API proxy-сервер для фронтенда

**Цель**: реализовать эндпоинты в подготовленном Express.js-сервере. Сервер выступает посредником (proxy) между фронтендом и внешним API — скрывает API-ключ на серверной стороне. Структура сервера, маршруты и базовая конфигурация уже описаны — нужно реализовать обработчики запросов, middleware и обработку ошибок.

**Чему научится студент**:

- Принципы REST — проектирование ресурсов, выбор HTTP-методов (GET, POST), коды ответов (200, 404, 500), структура URI
- Express.js — создание серверного приложения на Node.js, маршрутизация, middleware
- CORS (Cross-Origin Resource Sharing) — настройка разрешений для запросов из браузера с другого домена
- Rate-limiting — ограничение частоты запросов для защиты от злоупотреблений
- Документирование API — описание доступных эндпоинтов, параметров и форматов ответов

**Ожидаемый результат**: работающий proxy-сервер с 3 эндпоинтами (поиск пользователя, получение статистики, список доступных отчётов), настроенным CORS и rate-limiting, документацией эндпоинтов. Фронтенд-приложение использует этот сервер для загрузки данных.

### Тема 4 — Разработка клиентской части веб-приложения и реализация асинхронного взаимодействия с API

#### Задание 4.1 — Динамическая загрузка данных через API

**Цель**: перевести приложение с предзагруженных данных на динамическую загрузку через HTTP-запросы. Реализовать кастомный хук для загрузки данных с обработкой трёх состояний: загрузка, ошибка, данные получены. Интерфейс хука и компоненты-заглушки (спиннер загрузки, сообщение об ошибке) уже подготовлены — нужно реализовать логику внутри хука и подключить компоненты состояний.

**Чему научится студент**:

- Создание кастомных React-хуков для инкапсуляции логики
- Работа с Fetch API — отправка HTTP-запросов из браузера
- Жизненный цикл React-компонента: `useEffect` для выполнения побочных эффектов (side effects)
- Отмена запросов через `AbortController` при размонтировании компонента (предотвращение утечек памяти)
- Паттерн loading / error / data — стандартный подход к управлению состоянием асинхронных данных

**Ожидаемый результат**: приложение загружает данные асинхронно, показывает спиннер во время загрузки, корректно обрабатывает ошибки сети и отображает данные после успешной загрузки.

#### Задание 4.2 — Поиск пользователя в реальном времени с debounce

**Цель**: дописать логику поискового компонента в шапке приложения. Компонент с разметкой и стилями уже встроен — нужно реализовать: задержку перед отправкой запроса (debounce), сам запрос к API, отображение результатов поиска, обработку состояний (загрузка, пустой результат, ошибка).

**Чему научится студент**:

- Debounce-паттерн — задержка выполнения действия до окончания ввода пользователем (оптимизация количества запросов)
- Отмена устаревших запросов через `AbortController` — если пользователь продолжил ввод, предыдущий запрос отменяется
- Управление сложным UI-состоянием: одновременно отслеживать текст ввода, состояние загрузки, массив результатов и возможную ошибку
- Работа с таймерами в React (`setTimeout`, `clearTimeout` через `useRef`)

**Ожидаемый результат**: поле поиска в шапке приложения, которое при вводе 3+ символов отправляет запрос к API с задержкой 300мс и отображает найденных пользователей.

### Тема 5 — Реализация асинхронного взаимодействия клиентского приложения с серверным API и обеспечение динамического обновления пользовательского интерфейса

#### Задание 5.1 — Сравнение двух пользователей

**Цель**: реализовать логику параллельной загрузки данных двух пользователей и заполнить подготовленный шаблон сравнения. Пользователь вводит два никнейма, данные загружаются одновременно, результат отображается в виде визуального сопоставления (side-by-side). Вёрстка и карточки уже готовы — нужно реализовать асинхронную логику и обработку ошибок.

**Чему научится студент**:

- Параллельные асинхронные запросы через `Promise.all` и `Promise.allSettled`
- Обработка частичных ошибок — один пользователь загрузился, другой не найден (приложение не должно "падать")
- Работа с несколькими асинхронными потоками и синхронизация их результатов
- Визуальное отображение прогресса для каждого запроса отдельно

**Ожидаемый результат**: раздел "Сравнение", где видны ключевые метрики двух пользователей рядом (рейтинг, процент побед, эффективность, лучшие элементы), с корректной обработкой ситуаций, когда один из пользователей не найден.

---

#### Задание 5.2 — Автообновление данных с индикатором

**Цель**: реализовать хук для периодического обновления данных (polling) и компонент-индикатор обновления. Разметка индикатора подготовлена — нужно реализовать логику таймера, корректную очистку при размонтировании и подсветку изменившихся значений.

**Чему научится студент**:

- Работа с `setInterval` в React — правильная очистка (cleanup) в `useEffect` для предотвращения утечек
- Создание хуков с таймерами (`usePolling`) — инкапсуляция логики периодических запросов
- Визуализация "свежести" данных (stale/fresh state) — сколько времени прошло с последнего обновления
- Сравнение старых и новых данных для подсветки изменившихся значений

**Ожидаемый результат**: индикатор автообновления в шапке приложения (прогресс-бар до следующего обновления), кнопка ручного обновления, визуальная подсветка карточек и строк таблицы, значения которых изменились.

### Тема 6 — Unit-тесты и интеграционные тесты для ключевых компонентов на JavaScript

#### Задание 6.1 — Unit-тесты утилит и React-хуков

**Цель**: покрыть unit-тестами утилитарные функции (цветовая система статистик, вычисления показателей) и кастомные React-хуки (управление темой). Структура тестовых файлов и describe-блоки уже созданы — нужно написать конкретные test-кейсы.

**Чему научится студент**:

- Разница между unit-тестами (изолированное тестирование одной функции) и интеграционными тестами (проверка взаимодействия компонентов)
- Тестирование чистых функций — передать аргументы, проверить результат
- Параметризованные тесты (`test.each`) — один шаблон теста с множеством наборов входных/выходных данных
- Тестирование React-хуков через `renderHook` — вызов хука вне компонента для проверки его поведения

**Ожидаемый результат**: 20+ unit-тестов:

- Тесты цветовой системы — проверка всех порогов для каждого типа метрики
